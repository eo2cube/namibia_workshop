Global Waterpack Analysis - DataCube Implementation¶

SDGs4Namibia | Wednesday, 19 May 2021 | 2pm - 4pm CET¶


Background¶

This short notebook demonstrates the difference in workflow and amount of code for a remote sensing analysis, between a datacube approach versus a classic script approach on a local machine. In this case Michael Wang's (DLR) Global Waterpack Analysis (GWP), computed on his laptop, will be implemented here. The difference in lines of code will be presented during the session.

The GWP is used to derive data for SDG 6.1.

Data¶

The Global WaterPack is a MODIS derived product, produced by the German Aerospace Center (DLR). Two years (2013 and 2018) of daily datasets were ingested into the NamibiaCube, covering the whole country via three MODIS-tiles. All datasets contain binary values of 0 and 1:

  • 0 - no water (black pixel)
  • 1 - water (white pixel)

Figure 1: Three Global WaterPack tiles covering Namibia. Source: German Aerospace Center (DLR) - [GWP].

Decisions and Consequences¶

This session ties directly into the Ingestion session from Wednesday, 12 May 2021.
Consequences, Alternatives and Learned Lessons from preprocessing steps and YAML definitions will be presented here. The leading talking points will be:

  • consequence of the reprojection (EPSG:32733 - WGS 84 / UTM zone 33S) before Ingestion
  • induced 0's into the reprojection artefacts
  • artefacts in timeseries
  • YAML definition: choice of the NAs - is 255 a sensible value?

The way a DataCube is dealing with this¶

Instead of mosaicing all three tiles and then clipping them to the size of Namibia, the DataCube tiled all Global Waterpack scenes into predefined tiles (500 x 500 km). So only tiles that the shapefile is touching, will be loaded into the dataset. This is the very reason why the 'tiling-decision' is important during ingestion.

Smaller tiles (lets say 200 x 200 km) would've produced more entries into the database and used a slightly bit more storage space, but would also use less memory space, when loaded into the cube, covering only the necessary area.

Loading dataset¶

Up to this point, Michael used already about 90 lines of code in order to achieve a similar result the DataCube is managing with the dc.load() function alone. The only difference is the computation of square kilometers of water coverage per dataset, which is convinient to get out of the way at this point. Not doing it would result in producing even more redundant files and another round of file-management.

The majority of his code serves mostly file-management (read & write) of the GWP-GeoTiffs and their intermediate products he produces. Since his way mostly uses storage space and not memory, one can argue this way of analysis is also efficient (for a local machine). But the amount of work is significantly higher.

Lets plot the result and see for ourselves¶

Plotting the result reveals the problem, which goes way back to the ingestion process. Remember this YAML-snippet?

Figure 2: Namibia Workshop Slides - Day 1 | Afternoon Session. Source: check our [Github repository].

By definition during ingestion, all appearing 'no data' values will receive an assignment 255. The extend of the namibian shapefile includes areas which are not covered by MODIS-scenes. By default, this eastern strip will receive only values of 255.

Multiply them by 365 days / datasets per year and you'll get 90.075 - mystery solved.

Other ways¶

Since the 255 assignment is set in stone (at least until you reingest), there must be another way to solve this.

  • One possibility is to threshold every value, that exceeds 365. This way the yellow stripe becomes white - sort of NA.
  • It is also possible to map another value onto anything that is <= 254.
    Here we chose 0. Strictly seen, it is induced data, but since it is not contributing to the presence of water over Namibia, this is negligible.
In [11]:
dataset_resampled = dataset.where(dataset <= 254, 0)
water_2013 = dataset_resampled.resample(time='1Y').sum()
water_2013.GWP.plot(size=10, cmap="gnuplot")
Out[11]:
<matplotlib.collections.QuadMesh at 0x7f7a51992100>

First Result¶

This loos like a reasonable result. You will notice a red horizontal line in the left upper corner. The color code suggests a value somewhere around 180. This artefact is a remnant from a reprojection that was performed on the GWP-MODIS-Tiles before ingestion. Through a slight shift within the new projection, the stripe over the ocean received induced 0-values. So if you sum an overlapping area with 365 1-values and another 365 0-values on top, you will average out somewhere around 182.

((1 * 365) + (0 * 365)) / 2 = 182.5

Overall, the DataCube got to the same result - via a little detour - in only four functions, mostly one-liners. It took Michael over 100 lines of code in order to achieve the equivalent.
